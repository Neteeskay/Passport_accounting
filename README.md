# Passport Accounting

Информационная система паспортного учёта граждан Российской Федерации.

Проект реализован как клиент-серверное веб-приложение и предназначен для ведения карточек граждан, учёта паспортных данных, регистрации связанных записей и разграничения доступа между ролями сотрудников.

## О проекте

`Passport Accounting` — это учебный проект, моделирующий работу цифровой системы паспортного стола.  
Система позволяет хранить и обрабатывать карточки граждан, вести историю паспортов и сопутствующих отметок, администрировать пользователей и формировать печатные документы.

Проект ориентирован не только на реализацию CRUD-операций, но и на построение цельного продукта:

- с понятным интерфейсом;
- с разделением ролей;
- с централизованным хранением данных;
- с подготовкой карточек и реестра к выгрузке в PDF;
- с архитектурой, которую можно развивать дальше.

## Цель работы

Цель проекта — разработать прототип системы паспортного учёта, который демонстрирует:

- организацию клиент-серверного взаимодействия;
- проектирование и инициализацию базы данных;
- реализацию авторизации и ролевой модели;
- работу с предметной сущностью “карточка гражданина”;
- хранение связанных записей по разделам документа;
- экспорт данных в удобном для демонстрации и печати виде.

## Основные возможности

- авторизация пользователей по логину и паролю;
- разделение доступа между ролями `admin` и `operator`;
- создание, просмотр, редактирование и удаление карточек граждан;
- загрузка фотографии гражданина;
- фильтрация и поиск граждан по ключевым параметрам;
- подсчёт статистики по общему количеству записей и полу граждан;
- управление пользователями системы;
- работа со связанными разделами карточки:
  - регистрация;
  - дети;
  - брак;
  - воинская обязанность;
  - загранпаспорт;
  - смена ФИО;
  - история паспортов;
- экспорт карточки гражданина в PDF;
- экспорт реестра граждан в PDF.

## Роли пользователей

### `admin`

Администратор имеет доступ ко всем основным разделам системы и дополнительно может:

- просматривать список пользователей;
- создавать новых пользователей;
- редактировать пользователей;
- удалять пользователей;
- выполнять действия, доступные оператору.

### `operator`

Оператор работает с карточками граждан и связанными записями, но не имеет доступа к управлению пользователями.

## Архитектура решения

Проект разделён на две основные части:

- `frontend/` — пользовательский интерфейс и клиентская логика;
- `app/` — backend API, бизнес-логика и работа с базой данных.

### Общая схема

1. Пользователь входит в систему через форму авторизации.
2. Frontend получает токен и использует его при последующих запросах.
3. Backend проверяет токен, определяет роль пользователя и разрешает или запрещает доступ к маршруту.
4. Данные по гражданам хранятся в базе `SQLite`.
5. Карточка гражданина собирается backend из основной таблицы и связанных записей.
6. При необходимости backend формирует PDF-файл карточки или реестра.

## Структура backend

### `app/main.py`

Точка входа backend-приложения:

- создаёт FastAPI-приложение;
- подключает `CORS`;
- инициализирует базу данных;
- монтирует `/uploads`;
- подключает основной API-router.

### `app/api/routes`

HTTP-маршруты приложения:

- `auth.py` — авторизация и управление пользователями;
- `citizens.py` — работа с карточками граждан;
- `stamps.py` — работа с отдельными связанными записями;
- `system.py` — системные маршруты и проверка доступности.

### `app/services`

Сервисный слой:

- `auth.py` — токены, логин, logout, CRUD пользователей;
- `citizens.py` — основная логика карточки гражданина;
- `stamps.py` — низкоуровневый CRUD записей по секциям;
- `pdf_export.py` — генерация PDF;
- `system.py` — системные payload.

### `app/db`

- `schema.py` — схема таблиц и индексов;
- `session.py` — подключение к БД, включение `foreign_keys`, runtime-миграции.

## База данных

В проекте используются таблицы:

- `users` — пользователи системы;
- `auth_tokens` — выданные access token;
- `citizens` — основная карточка гражданина;
- `stamps` — связанные записи по разделам карточки.

### Почему используется единая таблица `stamps`

Вместо отдельной таблицы на каждый раздел карточки выбран единый универсальный механизм хранения. Это решение позволяет:

- не дублировать однотипную CRUD-логику;
- проще расширять систему новыми разделами;
- централизованно управлять связанными записями;
- собирать удобный frontend-формат уже на уровне сервисов.

## API и интеграция

Frontend работает с backend через REST API.

Ключевые маршруты:

### Auth

- `POST /api/v1/auth/login`
- `GET /api/v1/auth/me`
- `POST /api/v1/auth/logout`
- `GET /api/v1/auth/users`
- `POST /api/v1/auth/users`
- `PUT /api/v1/auth/users/{user_id}`
- `DELETE /api/v1/auth/users/{user_id}`

### Citizens

- `GET /api/v1/citizens`
- `GET /api/v1/citizens/stats`
- `GET /api/v1/citizens/{citizen_id}`
- `POST /api/v1/citizens`
- `PUT /api/v1/citizens/{citizen_id}`
- `DELETE /api/v1/citizens/{citizen_id}`
- `POST /api/v1/citizens/photo`
- `GET /api/v1/citizens/{citizen_id}/pdf`
- `GET /api/v1/citizens/registry/pdf`

### Stamps

- `GET /api/v1/citizens/{citizen_id}/stamps`
- `GET /api/v1/citizens/{citizen_id}/stamps/{stamp_id}`
- `POST /api/v1/citizens/{citizen_id}/stamps`
- `PUT /api/v1/citizens/{citizen_id}/stamps/{stamp_id}`
- `DELETE /api/v1/citizens/{citizen_id}/stamps/{stamp_id}`

Документация Swagger доступна после запуска backend:

```text
http://127.0.0.1:8000/docs
```

## Поиск и фильтрация граждан

Система поддерживает фильтрацию по:

- текстовому поиску;
- полу;
- дате рождения;
- серии и номеру паспорта;
- адресу регистрации.

Кроме списка карточек, backend возвращает статистику:

- общее число записей;
- количество мужчин;
- количество женщин.

## PDF-экспорт

Система формирует два вида PDF:

### Карточка гражданина

Содержит:

- основные персональные данные;
- паспортные данные;
- адрес регистрации;
- фото гражданина;
- связанные записи по разделам.

### Реестр граждан

Содержит:

- список граждан;
- ключевые поля карточек;
- пригодный для печати сводный документ.

## Форматы данных

Backend принимает даты:

- в ISO-формате `YYYY-MM-DD`;
- в формате `dd.mm.yyyy`.

Это позволяет frontend работать с привычным форматом отображения, а backend — с нормализованным хранением данных.

## Запуск проекта

### 1. Backend

Установить зависимости:

```bash
py -m pip install -r requirements.txt
```

Запустить backend:

```bash
py -m uvicorn app.main:app --reload
```

Backend будет доступен по адресу:

```text
http://127.0.0.1:8000
```

### 2. Frontend

Перейти в папку frontend:

```bash
cd frontend
```

Установить зависимости:

```bash
npm install
```

Запустить frontend:

```bash
npm run dev
```

По умолчанию frontend доступен по адресу:

```text
http://localhost:3000
```

### 3. Связь frontend и backend

По умолчанию frontend обращается к:

```text
http://127.0.0.1:8000
```

При необходимости можно задать:

```env
NEXT_PUBLIC_API_URL=http://127.0.0.1:8000
```
