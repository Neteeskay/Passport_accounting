# Backend: подробная шпаргалка для защиты

Этот документ нужен как подробный handover по backend-части проекта `Passport Accounting`.
Его можно отправить тем, кто делал backend, чтобы они могли уверенно отвечать на вопросы преподавателя:

- что именно реализовано;
- как устроена архитектура;
- почему выбраны такие решения;
- как работает база данных;
- как реализованы авторизация и роли;
- что делает каждый endpoint;
- какие ограничения и компромиссы есть в текущей версии.

---

## 1. Что делает backend в проекте

Backend отвечает за серверную часть системы учёта паспортных данных граждан. Его зона ответственности:

- хранение данных пользователей системы;
- хранение карточек граждан;
- хранение истории штампов, привязанных к карточке гражданина;
- авторизация пользователей по логину и паролю;
- разграничение прав между ролями `admin` и `operator`;
- выдача данных через HTTP API;
- экспорт карточки гражданина в PDF;
- начальная инициализация базы данных при запуске.

Проще говоря, backend принимает запросы от клиента, валидирует их, сохраняет данные в БД, проверяет права доступа и возвращает структурированные ответы.

---

## 2. Стек backend и почему он выбран

В проекте используется следующий стек:

- `Python`
- `FastAPI 0.115.0`
- `SQLite`
- стандартный модуль `sqlite3`
- `Pydantic`
- `Uvicorn 0.30.6`
- `reportlab 4.4.9`

### Почему выбран `FastAPI`

`FastAPI` подходит для учебного проекта по нескольким причинам:

- быстро поднимается с нуля;
- удобно описывать маршруты;
- есть встроенная валидация через `Pydantic`;
- автоматически доступна документация Swagger по адресу `/docs`;
- код получается достаточно компактным и читаемым.

### Почему выбрали `SQLite`

Для текущего этапа проекта выбрана `SQLite`, потому что:

- не требуется отдельный сервер БД;
- база хранится в одном файле;
- удобно запускать локально на любой машине;
- достаточно для учебного проекта и демонстрации CRUD-логики.

### Почему используется `sqlite3`, а не ORM

В проекте сознательно использован прямой доступ к БД через модуль `sqlite3`, а не ORM вроде `SQLAlchemy`.

Это решение оправдано для текущего масштаба, потому что:

- проще показать преподавателю реальную SQL-логику;
- меньше скрытой магии;
- легко контролировать SQL-схему и запросы;
- для небольшого проекта это быстрее и понятнее.

Если проект развивать дальше, можно перейти на ORM и миграции, но для текущей лабораторной прямой SQL является нормальным и прозрачным решением.

## 3. Общая структура backend

Backend разбит по слоям. Это важно, потому что преподаватель может спросить не только “что реализовано”, но и “как организован код”.

Структура каталогов:

```text
app/
  api/
    dependencies/
      auth.py
    routes/
      auth.py
      citizens.py
      stamps.py
      system.py
    router.py
  core/
    config.py
  db/
    schema.py
    session.py
  schemas/
    auth.py
    citizens.py
    system.py
  services/
    auth.py
    citizens.py
    pdf_export.py
    stamps.py
    system.py
  main.py
```

### Смысл каждого слоя

#### `app/main.py`

Это точка входа приложения.

Там:

- создаётся объект `FastAPI`;
- настраивается lifecycle приложения;
- при старте выполняется подготовка директорий и инициализация базы данных;
- подключается главный роутер API.

#### `app/api/routes`

Здесь описаны HTTP endpoint-ы.

Именно этот слой отвечает за:

- приём HTTP-запросов;
- чтение query/body/path параметров;
- вызов бизнес-логики из `services`;
- преобразование ошибок в HTTP-ответы (`404`, `409`, `401`, `403` и т.д.).

#### `app/api/dependencies`

Содержит зависимости FastAPI, связанные с авторизацией:

- извлечение Bearer-токена;
- получение текущего пользователя;
- проверка ролей.

Это помогает не дублировать одну и ту же логику в каждом роуте.

#### `app/services`

Это бизнес-логика.

Именно здесь:

- выполняются SQL-запросы;
- создаются и обновляются сущности;
- реализуются правила проверки данных;
- собираются ответы;
- выполняется экспорт PDF;
- реализуется авторизация и работа с токенами.

#### `app/schemas`

Это Pydantic-схемы для запросов и ответов.

Они нужны для:

- валидации входящих данных;
- стандартизации выходных данных;
- генерации документации в Swagger.

#### `app/db`

Этот слой отвечает за работу с базой данных.

Там:

- хранится SQL-схема таблиц;
- создаётся соединение с БД;
- включаются внешние ключи;
- выполняется начальная инициализация.

#### `app/core/config.py`

Хранит конфигурацию проекта:

- название проекта;
- версию;
- префикс API;
- режим debug;
- секрет для токенов;
- TTL токена;
- путь к БД;
- логины и пароли стартовых пользователей.

## 4. Жизненный цикл приложения

Очень важный вопрос для защиты: что происходит при запуске сервера.

### Порядок запуска

При старте приложения выполняется `lifespan` в `app/main.py`.

Внутри него вызываются:

1. `prepare_storage()`
2. `initialize_database()`

### Что делает `prepare_storage()`

Функция создаёт папку:

```text
storage/uploads
```

Она нужна для будущей работы с фотографиями граждан.

### Что делает `initialize_database()`

Функция:

1. получает соединение с SQLite;
2. выполняет все SQL-операторы создания таблиц и индексов;
3. вызывает `seed_default_users(connection)`;
4. делает `commit`.

То есть база не требует ручной инициализации до первого запуска. Если файл БД отсутствует, он будет создан автоматически.

### Где хранится база данных

По умолчанию используется:

```text
storage/passport_accounting.db
```

Если изменить переменную окружения `DATABASE_URL`, путь можно перенастроить.

Название API. Используется в приложении и документации.

Версия backend.

Префикс для всех основных роутов, например:

Включает debug-режим FastAPI.

Секретный ключ для подписи access token.

Ответ:

Время жизни токена в минутах. Сейчас по умолчанию `720`, то есть 12 часов.

#### `DEFAULT_ADMIN_*` и `DEFAULT_OPERATOR_*`

Это стартовые учётные записи, которые автоматически создаются при инициализации базы:

## 7. Проектирование базы данных

База данных состоит из четырёх таблиц:

- `users`
- `auth_tokens`
- `citizens`
- `stamps`

Это минимальная, но уже связанная схема.

### 7.1. Таблица `users`

Назначение: хранение учётных записей пользователей системы.

Поля:

- `id` — первичный ключ
- `username` — уникальный логин
- `password_hash` — хеш пароля
- `full_name` — имя пользователя
- `role` — роль (`admin` или `operator`)
- `is_active` — активна ли учётная запись
- `created_at` — дата создания
- `updated_at` — дата обновления

### Ограничения

- `username` уникален
- `role` ограничена через `CHECK (role IN ('admin', 'operator'))`
- `is_active` ограничен через `CHECK (is_active IN (0, 1))`

### Индексы

- индекс по `role`
- индекс по `is_active`

### Зачем это нужно

- быстро выбирать пользователей по роли;
- быстро фильтровать только активных пользователей;
- гарантировать корректные значения ролей и статуса.

---

### 7.2. Таблица `auth_tokens`

Назначение: хранение выданных access token и их статуса.

Поля:

- `id` — первичный ключ
- `token_id` — уникальный идентификатор токена (`jti`)
- `user_id` — ссылка на пользователя
- `expires_at` — timestamp истечения токена
- `revoked_at` — дата отзыва токена
- `created_at` — дата создания записи

### Связь

- `user_id -> users.id`
- при удалении пользователя действует `ON DELETE CASCADE`

### Индексы

- индекс по `user_id`
- индекс по `expires_at`

### Зачем нужна отдельная таблица токенов

Это важный момент для ответа преподавателю.

Если бы токены были полностью stateless, logout было бы сложно реализовать корректно: клиент удалил бы токен у себя, но сервер не знал бы, что токен надо больше не принимать.

Здесь выбран гибридный подход:

- токен подписан секретом и может проверяться по содержимому;
- при этом `jti` токена хранится в БД;
- сервер может отозвать токен через `revoked_at`.

То есть logout реализован как серверный отзыв токена.

---

### 7.3. Таблица `citizens`

Назначение: хранение основной карточки гражданина.

Поля:

- `id` — первичный ключ
- `last_name`
- `first_name`
- `middle_name`
- `birth_date`
- `passport_series`
- `passport_number`
- `issue_date`
- `issued_by`
- `registration_address`
- `notes`
- `photo_path`
- `created_by_user_id`
- `created_at`
- `updated_at`

### Связи и ограничения

- `created_by_user_id -> users.id`
- при удалении пользователя используется `ON DELETE SET NULL`
- пара `passport_series + passport_number` уникальна

### Почему `created_by_user_id` удаляется через `SET NULL`

Это правильное решение для истории данных.

Если пользователь-автор будет удалён, карточка гражданина не должна исчезнуть. Поэтому запись сохраняется, а ссылка на автора обнуляется.

### Индексы

- индекс по ФИО: `(last_name, first_name, middle_name)`
- индекс по `birth_date`
- индекс по паспортным данным: `(passport_series, passport_number)`

### Зачем это нужно

- ускорение поиска по имени;
- ускорение фильтрации по дате рождения;
- ускорение работы с паспортными данными;
- защита от дублирования одной и той же карточки.

---

### 7.4. Таблица `stamps`

Назначение: хранение истории штампов по гражданину.

Поля:

- `id` — первичный ключ
- `citizen_id` — ссылка на гражданина
- `stamp_type` — тип штампа
- `stamp_placed_at` — дата постановки
- `stamp_authority` — кем поставлен штамп
- `stamp_note` — примечание
- `created_at` — дата создания

### Связь

- `citizen_id -> citizens.id`
- при удалении гражданина действует `ON DELETE CASCADE`

### Почему здесь `CASCADE`

Штампы не являются самостоятельной сущностью без гражданина.
Если карточка гражданина удаляется, связанная история штампов тоже должна удалиться.

### Индексы

- индекс по `citizen_id`
- индекс по `stamp_placed_at`

### Зачем это нужно

- быстро получать все штампы гражданина;
- быстро сортировать или анализировать историю по датам.

## 9. Работа с соединением к БД

В `app/db/session.py` реализована низкоуровневая работа с SQLite.

### Как открывается соединение

Используется функция `get_connection()`.

Она:

- читает `DATABASE_URL`;
- преобразует путь формата `sqlite:///...`;
- создаёт папку для файла БД при необходимости;
- открывает соединение через `sqlite3.connect(...)`;
- включает `connection.row_factory = sqlite3.Row`;
- включает `PRAGMA foreign_keys = ON`.

### Зачем нужен `sqlite3.Row`

Чтобы обращаться к колонкам по имени, а не только по индексу.

То есть можно писать:

```python
row["username"]
```

а не:

```python
row[1]
```

### Зачем включать `PRAGMA foreign_keys = ON`

В SQLite внешние ключи по умолчанию могут не работать.
Эта команда явно включает контроль внешних ключей, чтобы:

- нельзя было привязать карточку к несуществующему пользователю;
- нельзя было создать штамп для несуществующего гражданина;
- работали `CASCADE` и `SET NULL`.

## 10. Авторизация: как она реализована

Это один из самых важных блоков для защиты.

### Общая идея

Авторизация состоит из нескольких этапов:

1. пользователь отправляет логин и пароль;
2. backend находит пользователя в БД;
3. backend проверяет пароль по хешу;
4. если всё верно, backend создаёт access token;
5. токен сохраняется в таблицу `auth_tokens`;
6. клиент использует этот токен в заголовке `Authorization: Bearer ...`.

### Хранение пароля

Пароль в БД хранится не в открытом виде, а в хешированном.

Используется:

- алгоритм `PBKDF2-HMAC-SHA256`
- `100000` итераций
- случайная соль

Формат хранения:

```text
pbkdf2_sha256$100000$<salt>$<digest>
```

### Почему это правильно

- пароль не хранится открытым текстом;
- даже если получить БД, пароль напрямую не виден;
- соль защищает от простых таблиц радужных хешей;
- большое число итераций усложняет перебор.

### Как проверяется пароль

Функция `verify_password(...)`:

1. разбирает строку хеша;
2. достаёт схему, число итераций, соль и ожидаемый digest;
3. повторно вычисляет хеш из введённого пароля;
4. сравнивает через `hmac.compare_digest(...)`.

### Почему используется `compare_digest`

Это безопаснее обычного сравнения строк, потому что снижает риск тайминговых атак.

## 11. Access token: как он устроен

Токен создаётся в `create_access_token(...)`.

### Состав токена

Токен имеет структуру:

```text
base64url(header).base64url(payload).base64url(signature)
```

То есть по форме он похож на JWT/JWS compact serialization.

### Что лежит в payload

- `jti` — уникальный id токена
- `sub` — id пользователя
- `username`
- `role`
- `iat` — время выдачи
- `exp` — время истечения

### Как подписывается токен

Подпись строится через:

- `HMAC`
- `SHA-256`
- секрет `AUTH_SECRET`

### Что важно сказать преподавателю

Это не библиотечная реализация JWT, а собственная компактная signed-token схема по тому же принципу:

- есть header;
- есть payload;
- есть подпись;
- сервер проверяет подпись и срок действия.

Для учебного проекта это допустимо и полезно, потому что показывает понимание механики токенов, а не только умение подключить готовую библиотеку.

## 12. Logout: как реализовано завершение сессии

Logout сделан не “на словах”, а реально на сервере.

### Как это работает

При логине:

- создаётся токен;
- его `jti` сохраняется в `auth_tokens`.

При logout:

- сервер декодирует токен;
- достаёт `jti`;
- ищет токен в БД;
- проставляет `revoked_at = CURRENT_TIMESTAMP`.

После этого токен считается отозванным и больше не должен приниматься.

### Почему это лучше, чем просто удалить токен на клиенте

Если просто удалить токен у клиента, но оставить его действительным на сервере, украденный токен всё ещё можно использовать.

Здесь сервер хранит состояние токена и умеет его отзывать.

### Какие ошибки возможны

- токен не передан;
- токен повреждён;
- токен неизвестен серверу;
- токен уже отозван;
- токен просрочен.

## 13. Проверка доступа и роли

В системе есть две роли:

- `admin`
- `operator`

### Где проверяется роль

Роль проверяется в dependency-слое через:

- `get_bearer_token`
- `get_current_auth_context`
- `get_current_user`
- `require_roles(...)`

### Как это работает пошагово

1. из заголовка извлекается Bearer-токен;
2. токен валидируется;
3. из токена и БД определяется текущий пользователь;
4. проверяется, входит ли его роль в разрешённый список.

### Где какие права

#### `admin`

Имеет доступ:

- ко всем рабочим роутам граждан и штампов;
- к `/api/v1/system/session`;
- к `/api/v1/system/admin`.

#### `operator`

Имеет доступ:

- к рабочим роутам граждан и штампов;
- к `/api/v1/system/session`;
- не имеет доступа к `/api/v1/system/admin`.

### Что возвращается при нарушении прав

- `401` — если токен отсутствует или невалиден;
- `403` — если пользователь аутентифицирован, но роль не подходит.

Это правильное разграничение с точки зрения HTTP-семантики.

## 14. Блок `system`: для чего он нужен

Системные endpoint-ы помогают:

- быстро проверить, что сервер запущен;
- получить базовую информацию о сервисе;
- проверить текущую сессию;
- продемонстрировать разграничение ролей.

### `GET /`

Возвращает краткую служебную информацию:

- название проекта;
- версию;
- API-префикс;
- путь к Swagger (`/docs`).

### `GET /health`

Возвращает простой health-check верхнего уровня.

### `GET /api/v1/system/health`

То же самое в пространстве versioned API.

### `GET /api/v1/system/info`

Возвращает информацию о сервисе уже как часть API.

### `GET /api/v1/system/session`

Это удобно для фронтенда и для проверки, что токен действительно распознан.

### `GET /api/v1/system/admin`

Admin-only endpoint.

Используется как демонстрация того, что backend реально различает роли.

## 15. Работа с карточками граждан

Карточка гражданина — центральная сущность проекта.

### 15.1. Создание карточки

Endpoint:

```text
POST /api/v1/citizens
```

Принимает:

- ФИО;
- дату рождения;
- паспортные данные;
- дату выдачи и кем выдан;
- адрес регистрации;
- заметки;
- путь к фотографии;
- id пользователя-создателя;
- список штампов.

### Что делает backend при создании

1. валидирует body через `Pydantic`;
2. нормализует строки: обрезает пробелы;
3. при необходимости проверяет, существует ли `created_by_user_id`;
4. вставляет запись в `citizens`;
5. вставляет связанные штампы в `stamps`;
6. делает `commit`;
7. возвращает созданную карточку целиком вместе со штампами.

### Какие ошибки предусмотрены

- `409 Conflict` — если уже существует гражданин с таким же паспортом;
- `404 Not Found` — если указан несуществующий `created_by_user_id`.

### Почему дубликат проверяется по серии и номеру паспорта

Паспортные данные выступают естественным идентификатором карточки в текущей модели.
Так система предотвращает создание дублей.

---

### 15.2. Редактирование карточки

Endpoint:

```text
PUT /api/v1/citizens/{citizen_id}
```

### Что меняется

Можно изменить:

- ФИО;
- дату рождения;
- паспортные данные;
- дату выдачи;
- орган выдачи;
- адрес регистрации;
- заметки;
- путь к фото;
- автора создания;
- весь список штампов.

### Как реализовано обновление штампов при редактировании

При обновлении гражданина backend:

1. обновляет основную карточку;
2. удаляет все старые штампы гражданина;
3. заново вставляет список штампов из запроса.

### Почему выбран такой подход

Для учебного проекта это проще и надёжнее:

- не нужно дифференцировать отдельно “какие штампы удалить, какие поменять, какие добавить”;
- состояние карточки после обновления полностью соответствует присланному payload.

Минус этого подхода:

- старые `id` штампов не сохраняются при полном обновлении карточки через `PUT`.

Если нужна более тонкая работа со штампами, для этого есть отдельные stamp-endpoint-ы.

### Какие ошибки предусмотрены

- `404 Not Found` — если гражданин не найден;
- `404 Not Found` — если не найден `created_by_user_id`;
- `409 Conflict` — если новые паспортные данные конфликтуют с другой записью.

---

### 15.3. Получение списка граждан

Endpoint:

```text
GET /api/v1/citizens
```

Поддерживает:

- поиск;
- фильтрацию;
- сортировку;
- пагинацию.

### Поддерживаемые query-параметры

- `search`
- `birth_date`
- `registration_address`
- `passport_series`
- `passport_number`
- `sort_by`
- `sort_order`
- `limit`
- `offset`

### По каким полям работает `search`

Поиск выполняется по:

- `last_name`
- `first_name`
- `middle_name`
- `passport_series`
- `passport_number`

### Какие варианты сортировки есть

Допустимые поля:

- `full_name`
- `birth_date`
- `created_at`
- `updated_at`
- `passport_series`

Направление:

- `asc`
- `desc`

### Как backend защищает сортировку

Это важный технический вопрос.

Поле сортировки не вставляется в SQL произвольно. В коде есть белый список `SORT_COLUMNS`.

То есть:

- если `sort_by` неизвестен, backend подставляет `full_name`;
- если `sort_order` неверный, backend подставляет `asc`.

Это снижает риск SQL-инъекции через блок сортировки.

### Что возвращается в списке

Для каждой карточки:

- базовые поля гражданина;
- `full_name`;
- `stamp_count`;
- служебные даты.

Отдельно возвращаются:

- `total`
- `limit`
- `offset`

Это удобно для фронтенда и пагинации.

### 15.4. Получение карточки в PDF

Endpoint:

```text
GET /api/v1/citizens/{citizen_id}/pdf
```

### Что делает backend

1. загружает карточку гражданина;
2. загружает связанные штампы;
3. передаёт данные в PDF-сервис;
4. генерирует PDF в памяти;
5. возвращает его как `application/pdf`.

### Что попадает в PDF

- ФИО;
- дата рождения;
- паспортные данные;
- дата выдачи;
- кем выдан;
- адрес регистрации;
- путь к фото;
- заметки;
- `created_by_user_id`;
- даты создания и обновления;
- история штампов.

### Работа с фото

Если у карточки есть `photo_path`, backend пытается найти файл:

- по абсолютному пути;
- относительно корня проекта;
- в `storage/uploads`;
- по нескольким разумным вариантам относительного расположения.

Если картинку открыть не удалось, PDF всё равно создаётся.
Фото считается необязательным элементом и не должно ломать экспорт.

## 16. Работа со штампами

Штампы вынесены в отдельный блок логики.

Это правильно архитектурно, потому что:

- гражданин — основная сущность;
- штампы — дочерняя сущность;
- у штампов есть собственный CRUD.

### 16.1. Получить список штампов

```text
GET /api/v1/citizens/{citizen_id}/stamps
```

Возвращает все штампы гражданина, отсортированные по:

- `stamp_placed_at ASC`
- затем `id ASC`

### 16.2. Получить один штамп

```text
GET /api/v1/citizens/{citizen_id}/stamps/{stamp_id}
```

Backend проверяет:

- существует ли гражданин;
- существует ли этот штамп у данного гражданина.

### 16.3. Создать штамп

```text
POST /api/v1/citizens/{citizen_id}/stamps
```

После создания:

- вставляется запись в `stamps`;
- у гражданина обновляется `updated_at`.

### Зачем обновлять `updated_at` у гражданина

Потому что изменение истории штампов фактически является изменением карточки гражданина как бизнес-сущности.

### 16.4. Обновить штамп

```text
PUT /api/v1/citizens/{citizen_id}/stamps/{stamp_id}
```

Обновляются:

- тип;
- дата;
- орган;
- примечание.

После обновления также меняется `updated_at` у гражданина.

### 16.5. Удалить штамп

```text
DELETE /api/v1/citizens/{citizen_id}/stamps/{stamp_id}
```

После удаления:

- запись удаляется из `stamps`;
- у гражданина обновляется `updated_at`.

### Какие ошибки предусмотрены

- `404`, если гражданин не найден;
- `404`, если штамп не найден у данного гражданина.

## 17. Валидация входных данных

Валидация выполняется через `Pydantic`-схемы.

### Примеры ограничений

#### Для штампа

- `stamp_type` — от 1 до 100 символов
- `stamp_authority` — от 1 до 255 символов
- `stamp_note` — до 1000 символов

#### Для гражданина

- фамилия, имя, отчество имеют ограничения по длине;
- серия и номер паспорта ограничены;
- `registration_address` обязателен;
- `notes` ограничены по длине;
- `photo_path` ограничен по длине;
- `created_by_user_id >= 1`.

### Нормализация

Кроме Pydantic-валидации, сервисный слой дополнительно:

- обрезает пробелы через `.strip()`;
- превращает пустые строки в `None` там, где это уместно;
- приводит даты к строковому виду для хранения в SQLite.

Это полезно, потому что снижает количество мусорных значений в базе.

## 18. Обработка ошибок

Backend использует понятную схему:

1. сервисный слой поднимает доменные исключения;
2. route-слой преобразует их в HTTP-коды.

### Примеры доменных ошибок

- `AuthenticationError`
- `InactiveUserError`
- `RevokedTokenError`
- `AuthorizationError`
- `CitizenConflictError`
- `RelatedUserNotFoundError`
- `CitizenNotFoundError`
- `StampNotFoundError`

### Почему это хороший подход

- бизнес-логика не привязана напрямую к FastAPI;
- сервисы можно тестировать отдельно;
- роуты остаются тонкими и читаемыми;
- HTTP-слой отвечает только за транспортное представление ошибок.

## Практические примеры API

Ниже приведены примеры, которые особенно полезно знать перед защитой.

### Пример 1. Логин

Запрос:

```http
POST /api/v1/auth/login
Content-Type: application/json

{
  "username": "admin",
  "password": "admin123"
}
```

Пример ответа:

```json
{
  "access_token": "<token>",
  "token_type": "bearer",
  "expires_in": 43200,
  "user": {
    "id": 1,
    "username": "admin",
    "full_name": "System Administrator",
    "role": "admin",
    "is_active": true,
    "created_at": "2026-05-11T10:00:00",
    "updated_at": "2026-05-11T10:00:00"
  }
}
```

Что важно объяснить:

- пароль не сравнивается напрямую, а сверяется по хешу;
- токен не только выдаётся клиенту, но и регистрируется в таблице `auth_tokens`;
- роль пользователя сразу попадает в payload токена и потом используется для access control.

### Пример 2. Получение текущего пользователя

Запрос:

```http
GET /api/v1/auth/me
Authorization: Bearer <token>
```

Ответ:

```json
{
  "id": 1,
  "username": "admin",
  "full_name": "System Administrator",
  "role": "admin",
  "is_active": true,
  "created_at": "2026-05-11T10:00:00",
  "updated_at": "2026-05-11T10:00:00"
}
```

Смысл этого endpoint-а:

- быстро проверить, что токен валиден;
- получить данные о текущем пользователе без повторного логина;
- дать фронтенду информацию о роли и профиле.

### Пример 3. Создание гражданина

Запрос:

```http
POST /api/v1/citizens
Authorization: Bearer <token>
Content-Type: application/json

{
  "last_name": "Иванов",
  "first_name": "Иван",
  "middle_name": "Иванович",
  "birth_date": "1990-05-12",
  "passport_series": "1234",
  "passport_number": "567890",
  "issue_date": "2010-06-01",
  "issued_by": "ОВД Центрального района",
  "registration_address": "г. Москва, ул. Примерная, д. 10",
  "notes": "Тестовая запись",
  "photo_path": "ivanov.jpg",
  "created_by_user_id": 1,
  "stamps": [
    {
      "stamp_type": "Регистрация",
      "stamp_placed_at": "2010-06-02",
      "stamp_authority": "МВД",
      "stamp_note": "Постоянная регистрация"
    }
  ]
}
```

Что происходит внутри:

1. body валидируется через `CitizenCreateRequest`;
2. строки нормализуются;
3. проверяется существование пользователя-автора;
4. создаётся строка в `citizens`;
5. создаются записи в `stamps`;
6. возвращается уже собранная карточка со штампами.

### Пример 4. Поиск и фильтрация граждан

Запрос:

```http
GET /api/v1/citizens?search=Иванов&sort_by=created_at&sort_order=desc&limit=20&offset=0
Authorization: Bearer <token>
```

Что покажет такой запрос:

- только подходящие записи;
- отсортирует их по времени создания от новых к старым;
- вернёт максимум 20 записей;
- пропустит 0 записей от начала выборки.

### Пример 5. Обновление гражданина

Запрос:

```http
PUT /api/v1/citizens/1
Authorization: Bearer <token>
Content-Type: application/json
```

Body по структуре совпадает с `POST /api/v1/citizens`.

Что важно помнить:

- `PUT` в этой реализации заменяет состояние карточки;
- список штампов после обновления соответствует именно присланному списку;
- старые штампы удаляются и вставляются заново.

### Пример 6. Добавление штампа отдельно

Запрос:

```http
POST /api/v1/citizens/1/stamps
Authorization: Bearer <token>
Content-Type: application/json

{
  "stamp_type": "Воинский учет",
  "stamp_placed_at": "2012-09-01",
  "stamp_authority": "Военкомат",
  "stamp_note": "Поставлен на учет"
}
```

Что происходит:

- backend убеждается, что гражданин существует;
- создаёт штамп;
- обновляет `updated_at` у гражданина;
- возвращает созданный штамп.

### Пример 7. Logout

Запрос:

```http
POST /api/v1/auth/logout
Authorization: Bearer <token>
```

Ответ:

```json
{
  "message": "User session has been successfully closed."
}
```

Что важно объяснить:

- logout выполняется на стороне сервера;
- токен помечается как отозванный;
- после этого доступ с ним должен быть запрещён.

### Типовые HTTP-статусы в проекте

- `200 OK` — успешное чтение или обновление
- `201 Created` — успешное создание
- `204 No Content` — успешное удаление штампа
- `401 Unauthorized` — нет токена, токен неверный, просрочен или отозван
- `403 Forbidden` — токен валиден, но роль не подходит
- `404 Not Found` — сущность не найдена
- `409 Conflict` — конфликт уникальности по паспортным данным

## 26. Частые вопросы преподавателя и готовые ответы

Ниже самые вероятные вопросы и варианты ответов.

### Вопрос: почему выбрали FastAPI?

Ответ:
потому что он хорошо подходит для учебного backend-проекта: даёт быструю разработку API, строгую валидацию через Pydantic и встроенную Swagger-документацию.

### Вопрос: почему SQLite, а не PostgreSQL?

Ответ:
на текущем этапе SQLite достаточно, потому что проект учебный, локальный и не требует отдельного сервера БД. Архитектурно модель уже реляционная, поэтому в будущем переход на PostgreSQL возможен.

### Вопрос: почему не использовали ORM?

Ответ:
для текущего масштаба проекта прямой SQL через `sqlite3` проще и прозрачнее. Это позволяет явно показать схему, связи, ограничения и запросы без скрытой магии ORM.

### Вопрос: как у вас хранится пароль?

Ответ:
не в открытом виде, а как хеш PBKDF2-HMAC-SHA256 с солью и 100000 итерациями.

### Вопрос: как работает login?

Ответ:
сервер ищет пользователя по логину, сверяет пароль по хешу, проверяет, что аккаунт активен, после чего создаёт подписанный access token и сохраняет его идентификатор в таблицу `auth_tokens`.

### Вопрос: как работает logout?

Ответ:
сервер не просто просит клиент удалить токен, а помечает токен в БД как отозванный через `revoked_at`. После этого даже при наличии токена на клиенте backend его больше не принимает.

### Вопрос: у вас JWT?

Ответ:
по структуре токен похож на JWT compact format: header, payload и signature в base64url. Но реализован вручную, чтобы показать механику подписи и валидации на учебном этапе.

### Вопрос: как проверяете права доступа?

Ответ:
через dependency-слой FastAPI. Сначала проверяется Bearer-токен, потом определяется текущий пользователь, потом сверяется его роль с разрешёнными ролями маршрута.

### Вопрос: какая разница между 401 и 403 у вас?

Ответ:
`401` возвращается, если пользователь не аутентифицирован: нет токена, токен невалиден, истёк или отозван. `403` возвращается, если пользователь уже аутентифицирован, но ему не хватает прав по роли.

### Вопрос: как предотвращаются дубли граждан?

Ответ:
на уровне БД стоит уникальное ограничение на пару `passport_series + passport_number`.

### Вопрос: почему штампы вынесены в отдельную таблицу?

Ответ:
потому что это отношение один-ко-многим: один гражданин может иметь несколько штампов. Нормализация в отдельную таблицу делает модель чище и гибче.

### Вопрос: почему при удалении гражданина штампы каскадно удаляются?

Ответ:
потому что штамп не имеет самостоятельного смысла без гражданина. Это зависимая сущность.

### Вопрос: почему при удалении пользователя карточка гражданина не удаляется?

Ответ:
потому что карточка гражданина — это самостоятельная бизнес-запись. Поэтому для `created_by_user_id` выбрано `ON DELETE SET NULL`.

### Вопрос: почему при изменении штампа обновляется `updated_at` у гражданина?

Ответ:
потому что история штампов является частью состояния карточки гражданина, и её изменение считается изменением самой карточки как бизнес-сущности.

### Вопрос: как сделан поиск?

Ответ:
через SQL `LIKE` по ФИО и паспортным полям, плюс отдельные фильтры по дате рождения, адресу и паспортным атрибутам.

### Вопрос: как вы защищаете сортировку от инъекций?

Ответ:
сортировка разрешена только по белому списку полей. Если пришло неизвестное поле, backend подставляет безопасное значение по умолчанию.

### Вопрос: как генерируется PDF?

Ответ:
через библиотеку `reportlab`. Backend собирает карточку гражданина и историю штампов, рендерит PDF в память и возвращает файл в HTTP-ответе.

### Вопрос: что будет, если фотографии нет?

Ответ:
экспорт не сломается. Фото необязательное. Если путь не найден или изображение не удалось открыть, PDF всё равно создаётся.

### Вопрос: что у вас создаётся при первом запуске?

Ответ:
создаётся SQLite-файл БД, создаются таблицы, индексы и два стартовых пользователя: admin и operator.

### Вопрос: почему использовали отдельный слой services?

Ответ:
чтобы не смешивать HTTP-логику и бизнес-логику. Routes остаются тонкими, а основная логика сосредоточена в сервисах и может переиспользоваться.

### Вопрос: что бы вы улучшили дальше?

Ответ:
добавили бы PostgreSQL, ORM или миграции, refresh token, управление пользователями, полноценные тесты и production-настройки безопасности.

## 28. Как лучше отвечать, если преподаватель начнёт “копать”

Хорошая схема ответа:

1. сначала назвать слой или сущность;
2. потом коротко описать её задачу;
3. потом привести конкретную реализацию из проекта.

Пример:

“Разграничение ролей сделано не прямо в каждом endpoint-е, а через dependency-слой FastAPI. Мы используем `require_roles(...)`, который сначала получает текущего пользователя из Bearer-токена, а потом проверяет, входит ли его роль в разрешённый список. Поэтому admin может открыть `/api/v1/system/admin`, а operator — нет”.

Это звучит намного сильнее, чем общий ответ без привязки к реализации.
