# T03. Сводка требований по SRS, Questions List и User Story

Этот документ фиксирует краткую сводку требований для проекта `Passport Accounting`.
Он подготовлен как рабочая выжимка для команды и может использоваться в `GitHub Wiki`, `README` и при постановке задач в `Issues`.

## 1. Назначение системы

`Passport Accounting` — это веб-приложение для учета паспортных данных граждан.
Система должна помогать сотрудникам хранить, просматривать, изменять и находить карточки граждан в одном месте без ведения разрозненных бумажных записей.

## 2. Источники требований

Сводка составлена по материалам:

- `SRS`, подготовленному в лабораторной работе №1
- `Questions List`
- `User Story`

## 3. Пользователи системы

В системе предусмотрены две основные роли:

- `Оператор`
- `Администратор`

### Оператор

Оператор работает с карточками граждан, вносит данные, редактирует записи, добавляет фотографии и штампы, выполняет поиск и подготавливает данные к печати.

### Администратор

Администратор имеет расширенные права: управляет учетными записями пользователей, контролирует доступ к разделам системы и выполняет резервное копирование и восстановление данных.

## 4. Основные цели проекта

- централизовать хранение паспортных данных граждан;
- упростить поиск и обновление записей;
- обеспечить разделение доступа по ролям;
- поддержать печать и экспорт карточек;
- сохранить данные при помощи архивации и резервного копирования.

## 5. Функциональные требования

## 5.1. Аутентификация и доступ

- пользователь должен входить в систему по логину и паролю;
- пользователь должен иметь возможность выйти из системы;
- при неверных учетных данных система должна показывать сообщение об ошибке;
- система должна разграничивать доступ по ролям `admin` и `operator`;
- защищенные разделы должны быть недоступны неавторизованным пользователям.

## 5.2. Работа с карточками граждан

- оператор должен иметь возможность создать новую карточку гражданина;
- оператор должен иметь возможность редактировать существующую карточку;
- система должна хранить основные персональные и паспортные данные;
- пользователь должен иметь возможность открыть детальную карточку гражданина;
- система должна поддерживать отображение списка граждан.

## 5.3. Паспортные и персональные данные

Для гражданина должны храниться:

- ФИО;
- дата рождения;
- серия и номер паспорта;
- дата выдачи паспорта;
- орган, выдавший паспорт;
- адрес регистрации;
- дополнительные служебные сведения и заметки.

## 5.4. Фотография гражданина

- система должна поддерживать загрузку фотографии гражданина;
- допустимые форматы: `JPG` и `PNG`;
- фотография должна отображаться в карточке гражданина;
- интерфейс должен поддерживать предварительный просмотр изображения.

## 5.5. Штампы

- система должна поддерживать хранение истории штампов;
- для каждого штампа должны фиксироваться тип, дата, орган и примечание;
- оператор должен иметь возможность добавлять и изменять штампы в карточке;
- история штампов должна отображаться в детальной карточке гражданина.

## 5.6. Поиск, фильтрация и сортировка

- система должна поддерживать поиск по ФИО;
- система должна поддерживать поиск по серии и номеру паспорта;
- система должна поддерживать поиск по адресу регистрации;
- система должна поддерживать поиск по дате рождения;
- список граждан должен поддерживать сортировку по ключевым полям;
- пользователь должен видеть результаты поиска в удобном списке.

## 5.7. Архивирование и восстановление

- удаление карточки не должно приводить к физической потере данных;
- запись должна переводиться в архив;
- пользователь должен иметь возможность восстановить запись из архива;
- система должна уметь показывать активные и архивные записи.

## 5.8. Печать и экспорт

- пользователь должен иметь возможность открыть печатную форму карточки;
- система должна поддерживать экспорт карточки гражданина в `PDF`;
- экспортируемый документ должен содержать основные сведения о гражданине и его штампах.

## 5.9. Управление пользователями

- администратор должен иметь возможность просматривать список пользователей;
- администратор должен иметь возможность создавать нового пользователя;
- администратор должен иметь возможность редактировать данные пользователя;
- администратор должен иметь возможность включать и отключать активность учетной записи.

## 5.10. Резервное копирование

- администратор должен иметь возможность экспортировать резервную копию данных;
- система должна поддерживать восстановление данных из резервной копии;
- для обмена резервными копиями может использоваться формат `JSON`.

## 5.11. Интерфейс

- интерфейс должен быть понятным для повседневной работы оператора;
- система должна поддерживать светлую и темную тему;
- формы должны показывать сообщения валидации при ошибках ввода.

## 6. Нефункциональные требования

## 6.1. Технологическая основа

- проект реализуется как веб-приложение;
- допустимый стек реализации: `Node.js`, `Express`, `EJS`, `SQLite`;
- приложение должно запускаться локально без сложной инфраструктуры.

## 6.2. Надежность и целостность

- данные граждан должны храниться последовательно и без потери связей;
- резервное копирование должно позволять восстановить рабочее состояние системы;
- архивирование должно сохранять запись для последующего восстановления.

## 6.3. Безопасность

- доступ к разделам системы должен контролироваться по ролям;
- неавторизованный пользователь не должен получать доступ к данным;
- пароль и вход в систему должны обрабатываться на стороне сервера.

## 6.4. Удобство использования

- основные сценарии должны выполняться без лишних переходов;
- формы должны быть пригодны для повседневной работы;
- поиск по карточкам должен сокращать время доступа к данным.

## 7. Основные сущности данных

Система оперирует следующими сущностями:

- `Пользователь`
- `Гражданин`
- `Штамп`
- `Резервная копия`

### Пользователь

Содержит логин, пароль, ФИО, роль и статус активности.

### Гражданин

Содержит персональные данные, паспортные сведения, адрес, заметки и ссылку на фотографию.

### Штамп

Связан с карточкой гражданина и содержит тип штампа, дату, орган и примечание.

## 8. Ключевые пользовательские сценарии

- вход в систему;
- создание карточки гражданина;
- редактирование карточки;
- поиск гражданина по параметрам;
- просмотр полной карточки;
- экспорт карточки в `PDF`;
- архивирование и восстановление записи;
- создание и восстановление резервной копии;
- управление пользователями.

## 9. Принятые рабочие допущения

- система разрабатывается как учебный локальный проект;
- хранилище данных может быть реализовано на `SQLite`;
- удаление записей реализуется как мягкое архивирование;
- резервное копирование реализуется через экспорт и импорт `JSON`;
- основной экспортный документ — карточка гражданина в `PDF`.
